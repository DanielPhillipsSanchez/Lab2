<task type="checkpoint:human-action" gate="blocking">
  <what-built>Claude started the dev server and tested the Snowflake connection via /api/metadata and a sample analytics endpoint.</what-built>
  <how-to-verify>
    Review the curl output from Task 2.

    SCENARIO A — Connectivity confirmed (metadata returned domain list):
    Go to Snowflake Web UI and run the date range diagnostic query to find what dates have actual MARTS data:

    ```sql
    SELECT
      'AUTHORIZATIONS' as table_name, MIN(transaction_date) as min_date, MAX(transaction_date) as max_date, COUNT(*) as total_rows
    FROM COCO_SDLC_HOL.MARTS.AUTHORIZATIONS
    UNION ALL SELECT 'SETTLEMENTS', MIN(settlement_date), MAX(settlement_date), COUNT(*) FROM COCO_SDLC_HOL.MARTS.SETTLEMENTS
    UNION ALL SELECT 'DEPOSITS', MIN(deposit_date), MAX(deposit_date), COUNT(*) FROM COCO_SDLC_HOL.MARTS.DEPOSITS
    UNION ALL SELECT 'CHARGEBACKS', MIN(chargeback_date), MAX(chargeback_date), COUNT(*) FROM COCO_SDLC_HOL.MARTS.CHARGEBACKS
    UNION ALL SELECT 'RETRIEVALS', MIN(retrieval_date), MAX(retrieval_date), COUNT(*) FROM COCO_SDLC_HOL.MARTS.RETRIEVALS
    UNION ALL SELECT 'ADJUSTMENTS', MIN(adjustment_date), MAX(adjustment_date), COUNT(*) FROM COCO_SDLC_HOL.MARTS.ADJUSTMENTS;
    ```

    Note the date ranges returned. The date filter in the portal must overlap with these ranges for KPIs to show non-zero values.

    SCENARIO B — SNOWFLAKE_NOT_CONFIGURED or connection error:
    1. Stop the dev server (Ctrl+C in the terminal running npm run dev)
    2. Fix .env.local with correct credentials
    3. Restart: cd apps/frontend && npm run dev
    4. Re-run: curl -s "http://localhost:3000/api/metadata"
    5. Repeat until connectivity is confirmed

    SCENARIO C — Connection works but authorization/kpis returned zeros:
    This is the date range mismatch. The date range diagnostic SQL above will reveal the correct dates.
  </how-to-verify>
