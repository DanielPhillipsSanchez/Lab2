Apply all three changes to all 5 authorization route files. Read each file before modifying it.

**Three changes per file:**

**Change 1 — Config import:** Add `import { FULL_TABLE_AUTHORIZATIONS } from '@/lib/config';` to the import section. Replace every occurrence of the hardcoded string `COCO_SDLC_HOL.MARTS.AUTHORIZATIONS` in SQL templates with `${FULL_TABLE_AUTHORIZATIONS}`.

**Change 2 — Error sanitization:** In every catch block, remove the `details: String(error),` line. The console.error call above it stays. The returned JSON keeps success, error, message, and code fields — just not details.

**Change 3 — Parameterized queries:** For each route, identify every value that comes from `searchParams.get()` and is currently interpolated into the SQL string. Build a binds array and replace the interpolated value with a `?` placeholder.

**Parameterization rules:**
- `startDate` and `endDate` are always present — always the first two binds: `const binds: (string | number | null)[] = [startDate, endDate];`
- Optional filters like `cardBrand` and `status`: build the filter string and push to binds in the same conditional block:
  ```typescript
  let cardBrandFilter = '';
  if (cardBrand) {
    cardBrandFilter = 'AND card_brand = ?';
    binds.push(cardBrand);
  }
  ```
- `limit` and `offset` in details routes: keep as validated integer literals. Parse them with parseInt() and inject into SQL as `LIMIT ${limit} OFFSET ${offset}` — NOT as bind parameters (Snowflake does not support parameterized LIMIT/OFFSET)
- `status` in details route maps to string values 'Approved'/'Declined' — parameterize the value, not the column:
  ```typescript
  // Before: if (status === 'approved') whereClause += ` AND approval_status = 'Approved'`;
  // After:
  if (status === 'approved') {
    whereClause += ' AND approval_status = ?';
    binds.push('Approved');
  } else if (status === 'declined') {
    whereClause += ' AND approval_status = ?';
    binds.push('Declined');
  }
  ```
- Pass binds to executeQuery: `const result = await executeQuery(sql, binds);`

**WHERE clause pattern for details route — current code uses string concatenation, preserve that pattern but with ? placeholders:**
```typescript
// Before (details route):
let whereClause = `WHERE transaction_date BETWEEN '${startDate}' AND '${endDate}'`;
if (cardBrand) whereClause += ` AND card_brand = '${cardBrand}'`;
if (status === 'approved') whereClause += ` AND approval_status = 'Approved'`;

// After:
const binds: (string | number | null)[] = [startDate, endDate];
let whereClause = 'WHERE transaction_date BETWEEN ? AND ?';
let cardBrandFilter = '';
if (cardBrand) {
  cardBrandFilter = ' AND card_brand = ?';
  binds.push(cardBrand);
}
let statusFilter = '';
if (status === 'approved') {
  statusFilter = ' AND approval_status = ?';
  binds.push('Approved');
} else if (status === 'declined') {
  statusFilter = ' AND approval_status = ?';
  binds.push('Declined');
}
```

Read each file individually before modifying to understand its exact structure. The timeseries, by-brand, and declines routes may have different filter structures — apply the same principles.
